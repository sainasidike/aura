# Aura 排盘精度技术文档

## 概述

Aura 的排盘系统由三大引擎组成，共享同一条**真太阳时标准化管线**，确保三大体系的时辰判定完全一致。

```
用户输入(钟表时间) → 时间标准化管线 → 真太阳时 + 时辰
                                          ↓
                          ┌────────────────┼────────────────┐
                          ↓                ↓                ↓
                     八字引擎         紫微斗数引擎       星盘引擎
                   (lunar-javascript) (fortel-ziweidoushu) (astronomy-engine)
```

---

## 一、时间标准化管线

> 源码：`src/lib/time/solar-time.ts`

**为什么需要真太阳时？**
中国命理以"时辰"为最小时间单位（2小时一个），而钟表时间是基于标准时区的均匀时间。同一时刻，不同经度的真太阳时不同。差异可达 ±30 分钟，足以改变时辰判定，从而完全改变排盘结果。

### 1.1 完整管线

```
钟表时间 → ① 夏令时检测 → ② UTC 转换 → ③ 儒略日计算
         → ④ 均时差计算 → ⑤ 经度修正 → ⑥ 真太阳时 → ⑦ 时辰判定
```

### 1.2 各步骤详解

#### ① 夏令时检测 (China DST 1986-1991)

中国仅在 1986-1991 年实行过夏令时（UTC+9），精确日期来自 IANA tzdata：

| 年份 | 开始日期 | 结束日期 |
|------|---------|---------|
| 1986 | 5月4日  | 9月14日 |
| 1987 | 4月12日 | 9月13日 |
| 1988 | 4月17日 | 9月11日 |
| 1989 | 4月16日 | 9月17日 |
| 1990 | 4月15日 | 9月16日 |
| 1991 | 4月14日 | 9月15日 |

- 夏令时期间 UTC 偏移量从 +8 变为 +9
- 函数：`isChinaDST(year, month, day, hour, timezone)`

#### ② UTC 转换

根据时区和夏令时状态确定 UTC 偏移量：

| 时区 | 标准偏移 | 夏令时偏移 | 标准经线 |
|------|---------|-----------|---------|
| Asia/Shanghai | UTC+8 | UTC+9 (仅1986-1991) | 120°E |
| Asia/Urumqi | UTC+6 | 无 | 90°E |
| Asia/Hong_Kong | UTC+8 | 无 | 120°E |
| Asia/Taipei | UTC+8 | 无 | 120°E |

#### ③ 儒略日计算

算法来源：Jean Meeus《Astronomical Algorithms》第7章

```
JD = INT(365.25 × (Y + 4716)) + INT(30.6001 × (M + 1)) + D + H/24 + B - 1524.5
```

其中 B = 2 - INT(Y/100) + INT(Y/400)，处理格里高利历修正。

#### ④ 均时差 (Equation of Time)

算法来源：Meeus 第28章简化版，纯 JS 实现，精度约 ±1 秒。

均时差是真太阳时与平太阳时之间的差异，由地球轨道离心率和黄道倾角共同造成，全年在 -14 分钟到 +16 分钟之间波动。

关键参数：
- **太阳几何平黄经** L₀ = 280.46646 + 36000.76983T
- **太阳平近点角** M = 357.52911 + 35999.05029T
- **黄道倾角** ε = 23.439291 - 0.013004T
- **离心率** e = 0.016708634 - 0.000042037T

计算公式：
```
EoT = y²sin(2L₀) - 2e·sin(M) + 4ey²sin(M)cos(2L₀) - 0.5y⁴sin(4L₀) - 1.25e²sin(2M)
```
其中 y = tan(ε/2)，结果单位为弧度，乘以 229.18 转换为分钟。

**精度验证**：与 Swiss Ephemeris (NASA 级精度库) 对比，偏差在 ±0.02 分钟以内。

#### ⑤ 经度修正

```
经度修正（分钟） = (出生地经度 - 标准经线) × 4
```

- 标准经线：Asia/Shanghai → 120°E，Asia/Urumqi → 90°E
- 每 1° 经度 = 4 分钟时间差
- 例：成都(104.07°E) → (104.07 - 120) × 4 = **-63.72 分钟**

#### ⑥ 真太阳时

```
真太阳时 = 钟表时间 + 经度修正 + 均时差
```

总修正量 = 经度修正 + 均时差（典型范围：-65 到 +30 分钟）

#### ⑦ 时辰判定

| 时辰 | 真太阳时范围 | 地支 |
|------|-------------|------|
| 子时 | 23:00 - 01:00 | 子 |
| 丑时 | 01:00 - 03:00 | 丑 |
| 寅时 | 03:00 - 05:00 | 寅 |
| 卯时 | 05:00 - 07:00 | 卯 |
| 辰时 | 07:00 - 09:00 | 辰 |
| 巳时 | 09:00 - 11:00 | 巳 |
| 午时 | 11:00 - 13:00 | 午 |
| 未时 | 13:00 - 15:00 | 未 |
| 申时 | 15:00 - 17:00 | 申 |
| 酉时 | 17:00 - 19:00 | 酉 |
| 戌时 | 19:00 - 21:00 | 戌 |
| 亥时 | 21:00 - 23:00 | 亥 |

**边界预警**：当真太阳时距时辰边界 < 15 分钟时，系统会发出警告，建议用户参考相邻两个时辰的结果。

## 三、紫微斗数引擎

> 源码：`src/lib/engines/ziwei.ts`
> 依赖：`fortel-ziweidoushu@1.3.4` + `opencc-js@1.0.5`

### 3.1 计算流程

```
真太阳时时辰 → 繁体时辰名映射 → fortel DestinyBoard → toJSON()
             → opencc 繁简转换 → 四化提取 → 十二宫完整数据
```

### 3.2 关键技术细节

**子时特殊处理**：fortel-ziweidoushu 不接受通用的"子時"，必须区分：
- `早子時` — 00:00-01:00（真太阳时 hour < 1）
- `夜子時` — 23:00-00:00（真太阳时 hour >= 23）

**繁简转换**：fortel 输出为繁体中文，使用 opencc-js (tw → cn) 转换。注意 fortel 的 `toJSON()` 返回的不是纯字符串，必须先 `String()` 再传入 opencc。

**四化提取**：toJSON() 不包含四化信息，必须通过 board 活对象获取：
```javascript
board.cells.forEach(cell => {
  cell.majorStars.forEach(star => {
    const derivative = board.getMajorStarDerivative(star); // 祿/權/科/忌
  });
});
```

### 3.3 输出数据

| 字段 | 说明 | 示例 |
|------|------|------|
| destinyMaster | 命主星 | 破军 |
| bodyMaster | 身主星 | 火星 |
| element | 五行局 | 土五局 |
| cells[12] | 十二宫 | 每宫含地支、宫名、主星、辅星、四化 |

每宫(cell)结构：
```json
{
  "ground": "子",
  "temples": ["命宫"],
  "majorStars": [{"name": "紫微", "fourInfluence": "化权"}],
  "minorStars": [{"name": "天魁"}, {"name": "恩光"}]
}
```

## 四、西洋星盘引擎

> 源码：`src/lib/engines/astrology.ts`
> 依赖：`astronomy-engine@2.1.19`（纯 JS，NASA JPL 精度）

### 4.1 计算流程

```
UTC 时间 → astronomy-engine MakeTime()
         → 行星位置(黄经/黄纬) → 星座/度数
         → ASC 计算 → 等宫制 → 宫位分配 → 相位计算
```

### 4.2 行星计算

| 天体 | 计算方法 | 精度 |
|------|---------|------|
| 太阳 | `Astronomy.SunPosition(time)` | 角秒级 |
| 月亮 | `Astronomy.GeoMoon(time)` + `Ecliptic()` | 角秒级 |
| 水金火木土 | `Astronomy.GeoVector(body, time, true)` + `Ecliptic()` | 角秒级 |
| 天海冥 | 同上 | 角分级 |

**逆行检测**：比较当日与前一日黄经差，若差值 < 0 则标记为逆行。

### 4.3 ASC (上升点) 计算

使用简化 Placidus 算法：

```
GMST = 280.46061837 + 360.98564736629 × (JD - 2451545.0) + ...
LST = GMST + 出生地经度
ASC = atan2(-cos(LST), sin(LST)×cos(ε) + tan(φ)×sin(ε))
```

其中 ε = 黄道倾角，φ = 出生地纬度。

### 4.4 宫位系统

使用**等宫制 (Equal House)**：从 ASC 起，每 30° 一宫，共 12 宫。

MC (天顶) 简化为 ASC + 270°。

### 4.5 相位计算

| 相位 | 角度 | 容许度(orb) |
|------|------|------------|
| 合相 | 0° | ±8° |
| 六合 | 60° | ±6° |
| 四分 | 90° | ±7° |
| 三合 | 120° | ±7° |
| 对冲 | 180° | ±8° |

### 4.6 输出数据

```json
{
  "planets": [
    {"name": "太阳", "sign": "双子", "degree": 23, "minute": 52, "house": 10, "retrograde": false},
    {"name": "月亮", "sign": "双鱼", "degree": 11, "minute": 45, "house": 7, "retrograde": false}
  ],
  "houses": [{"number": 1, "sign": "处女", "degree": 15, "minute": 30, "longitude": 165.5}],
  "aspects": [{"planet1": "太阳", "planet2": "月亮", "type": "四分", "angle": 90, "orb": 2.13}],
  "ascendant": 165.5,
  "midheaven": 75.5
}
```

## 六、精度边界与已知限制

| 项目 | 精度 | 说明 |
|------|------|------|
| 均时差 | ±0.02 分钟 | 与 Swiss Ephemeris 对比验证 |
| 行星黄经 | 角秒级 | astronomy-engine 基于 NASA JPL 星历表 |
| 经度修正 | 精确 | 纯数学换算，无近似 |
| 夏令时 | 精确到日 | 基于 IANA tzdata，不含小时级边界 |
| ASC 计算 | ~0.5° | 简化算法，非完整 Placidus |
| 宫位 | 等宫制 | 非 Placidus，高纬度地区差异较大 |
| 农历/节气 | lunar-javascript 内置 | 经广泛验证的开源库 |

### 可能影响结果的边界情况

1. **夏令时日边界**：夏令时起止日当天凌晨 2:00 前后出生，时钟拨动可能导致歧义
2. **时辰边界**：距时辰边界 < 15 分钟时，系统会自动预警
3. **子时换日**：默认使用子正换日(00:00)，部分流派使用子初换日(23:00)
4. **极端经度**：中国最西端(喀什 75.99°E)与标准经线差 44°，经度修正达 -176 分钟
